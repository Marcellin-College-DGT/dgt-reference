Testing & evidence

The part of the standard that moves a program from working to Excellence — checking input, handling the cases that break things, and proving you did.

The tkinter reference shows you how to build a program. This one shows you how to make it good — and, just as importantly, how to prove it is. For AS91896 (Year 12) and AS91906 (Year 13), the code that "works" earns Achieved. The marks above that come almost entirely from testing, validation, and the evidence you show for both.

Here is the uncomfortable truth most ākonga learn too late: at this level the difference between Achieved and Excellence is rarely the cleverness of your code. It is whether you checked that bad input is rejected, tested the values on the edges, handled the cases that would otherwise crash your program — and documented all of it clearly. This guide is about that work.

How each page is laid out

  • What it is — the idea in a sentence or two.
  • Why it matters — how it maps onto the grade you can earn.
  • Example — a complete program you can run to see it work.
  • In your assessment — exactly what to put in your evidence.
  • Common gotcha — the mistake that quietly caps your grade.

Running the demos. Every example links to a Python file. These are console programs (no window), so run them in VS Code or from a terminal with, for example, python validation_demo.py, and read the printed output. The testing ideas are the same whether your real program takes input from input() or from a tkinter Entry — only the source of the value changes, never the checking.

The ideawhy testing is where the grade lives

Where the marks areL2L3

The grade ladder, read as a testing ladder.

What it is

Both programming standards describe three grades. If you line them up, they are really describing three depths of testing and robustness. Read them this way and the path to Excellence stops being mysterious.

Achieved — it works

The program performs the task and runs on ordinary, expected input. It is documented with comments. That is the whole bar — and it is also the ceiling if you stop here.

Merit — informed

Add organised testing on expected and boundary cases, comments that describe what the code does and why, and code that follows the conventions of the language. The program is now tested at its edges, not just its middle.

Excellence — refined

The program is well-structured, flexible, and robust: it checks input for validity and correctly handles expected, boundary and invalid cases. It is comprehensively tested and debugged, and you can show that journey.

The one-sentence version

Expected cases pass → Achieved. Boundary cases tested too → Merit. Invalid cases handled and the whole thing proven robust → Excellence.

Common gotcha

Students pour their time into making the code do something impressive, then submit with no real testing. A simple program, thoroughly tested and robust, will out-grade a clever program that falls over on an empty input box.

Expected, boundary, invalidL2L3

The three kinds of test case — and why each earns its grade.

What it is

Every input to your program falls into one of three groups. Naming them is what lets you say, truthfully, "I tested expected, boundary, and invalid cases" — the exact words the grade descriptions use.

  • Expected — ordinary valid input from the middle of the allowed range. The program is built for these. e.g. a mark of 72 out of 100.
  • Boundary — the values right on the edges of what is allowed, where off-by-one bugs live. e.g. 0, 100, and each grade threshold like 50 and 80.
  • Invalid — input that must be rejected, not processed. e.g. -1, 101, or the word "abc".

Example

This program grades a mark out of 100, then runs all three kinds of case so you can see the difference: 

LOWEST = 0
HIGHEST = 100

def grade_mark(mark):
    if mark < LOWEST or mark > HIGHEST:
        raise ValueError(f"mark must be {LOWEST}-{HIGHEST}, got {mark}")
    if mark >= 80:
        return "Excellence"
    if mark >= 60:
        return "Merit"
    if mark >= 50:
        return "Achieved"
    return "Not Achieved"

# expected (normal), boundary (edges), invalid (must be rejected)
print(grade_mark(72))     # Merit
print(grade_mark(80))     # Excellence  -- a boundary worth checking
print(grade_mark(101))    # raises ValueError -- correctly rejected

In your assessment

For each feature of your program, list a few inputs of each kind and what should happen. Boundary cases are the ones markers look for, because they are the ones students forget — always test the lowest value, the highest value, and any threshold where behaviour changes.

Common gotcha

Testing only expected cases. Three "normal" inputs that all pass proves almost nothing — of course the happy path works. The marks are in the edges and the rejections.

Making it robustthe work that earns Excellence

Validating inputL2L3

Never trust what the user typed.

What it is

Checking that input is sensible before you use it. A program that assumes the user always types a valid number is fragile — it crashes the moment they don't. Checking input for validity is named directly in both standards as part of being "flexible and robust".

Why it matters

This is the single highest-value habit for your grade. An unvalidated program tops out at Achieved because it cannot handle invalid cases. Add validation and you unlock the Excellence criteria.

Example

The same task written twice — fragile, then robust:

Fragile — crashes on bad input

def fragile_age(raw):
    age = int(raw)          # crashes on "abc" or "" with a ValueError
    return f"Next year you will be {age + 1}"

Robust — checks type and range first

MIN_AGE = 0
MAX_AGE = 130

def robust_age(raw):
    raw = raw.strip()
    if not raw.isdigit():                 # rejects "", "abc", "-5", "3.5"
        return "Please enter your age as a whole number."
    age = int(raw)
    if age < MIN_AGE or age > MAX_AGE:    # rejects out-of-range numbers
        return f"Age must be between {MIN_AGE} and {MAX_AGE}."
    return f"Next year you will be {age + 1}"

In your assessment

Validate every place the user gives you data. In a tkinter program that means checking entry.get() before you use it — the same logic as above, just a different source. Show a test where you type rubbish into a box and the program responds with a clear message instead of crashing.

Common gotcha

Using int(entry.get()) directly. An empty box or a typo throws an uncaught error and your program dies in front of the marker. Check first, convert second.

Handling the invalid caseL2L3

Rejecting bad input gracefully, not crashing.

What it is

Validation spots bad input; handling it well means responding in a way that keeps the program running and tells the user what to fix. "Correctly handling expected, boundary and invalid cases" is the exact phrase from the Excellence criteria.

Why it matters

A crash is a failure to handle the invalid case. Catching it and showing a clear message is the difference between a program that "mostly works" and one that is genuinely robust.

Example

Two clean ways to handle invalid input. Choose whichever fits — return a message, or guard with try/except when a conversion might fail: 

from tkinter import messagebox

def add_score(raw, scores):
    raw = raw.strip()
    if not raw.isdigit():
        messagebox.showwarning("Invalid", "Scores must be whole numbers.")
        return                      # stop here -- do NOT add bad data
    scores.append(int(raw))

# When a conversion itself might fail, guard it:
def safe_int(raw):
    try:
        return int(raw)
    except ValueError:
        return None                 # caller checks for None and reacts

In your assessment

For each input, show what happens when it is invalid: the message shown, the data not stored, the program carrying on. That screenshot or test-table row is direct evidence of handling the invalid case.

Common gotcha

Wrapping the whole program in one giant try/except that silently swallows everything. That hides bugs rather than handling them. Catch the specific error, close to where it happens, and respond clearly.

Flexible code, not magic numbersL2L3

Use named constants and derived values in place of literals.

What it is

A "magic number" is a bare value like 500 or 0.15 sitting in your code with no name. Both standards reward "using constants, variables and derived values in place of literals" — giving values names and working things out from them rather than hard-coding.

Why it matters

It is half of what "flexible" means in the grade criteria. Named constants make a program easier to read, easier to change, and harder to get subtly wrong — and markers can see the intent at a glance.

Example

Magic numbers — what does 500 mean? why 0.15?

def total(price, qty):
    if qty > 10:
        price = price * 0.85        # 0.85? a discount? how much?
    return price * qty * 1.15       # 1.15? tax? which rate?

Named — the code now explains itself

GST_RATE = 0.15
BULK_THRESHOLD = 10
BULK_DISCOUNT = 0.15

def total(price, qty):
    if qty > BULK_THRESHOLD:
        price = price * (1 - BULK_DISCOUNT)   # derived, not hard-coded
    return price * qty * (1 + GST_RATE)

In your assessment

Put your fixed values in clearly named constants near the top of the file or class. When you explain your code, point to one and note that changing it in a single place updates the whole program — that is flexibility you can articulate.

Common gotcha

The same literal repeated in several places. If you ever change one and miss another, the program is now inconsistent. One named constant, used everywhere, removes that whole class of bug.

Evidencing itproving the testing happened

The test tableL2L3

The single most useful piece of evidence you can produce.

What it is

A simple table listing each test: what you fed in, what kind of case it is, what you expected, what actually happened, and whether it passed. It turns "I tested it" into something a marker can see and trust.

Why it matters

"Organised testing" is the wording for Merit; "comprehensively tested" is the wording for Excellence. A test table is what organised and comprehensive look like on the page. Without it, your testing is invisible — and invisible testing earns no marks.

Example

One feature — adding a score that must be 0–100 — tested across all three kinds of case:

Input Case Expected Actual ✓/✗
72 expected added added ✓
0 boundary added added ✓
100 boundary added added ✓
101 invalid rejected, message rejected, message ✓
abc invalid rejected, message rejected, message ✓
(empty) invalid rejected, message rejected, message ✓

In your assessment

Make one of these for each feature. Where a test fails, do not hide it — record the failure, fix the bug, and add a row showing it now passes. A fixed failure is stronger evidence than a row that was green from the start, because it proves you tested and refined.

Common gotcha

Writing the table from memory after you finish, with every row already passing. Markers can tell. Build it as you go, edges and invalid inputs included.

An automatic test runnerL3

Let the program test itself and print the report.

What it is

Instead of testing by hand each time, you write your cases once as data and loop over them, printing PASS or FAIL. You do not need a testing library — a list of cases and a loop is clearer for a marker to read, and re-runs in a second every time you change your code.

Why it matters

This is organised, repeatable, comprehensive testing — and the printed report drops straight into your evidence. It also catches the bug you accidentally reintroduce later, which hand-testing misses.

Example

def final_price(price, discount_percent):
    if price < 0:
        raise ValueError("price cannot be negative")
    if not 0 <= discount_percent <= 100:
        raise ValueError("discount must be 0-100")
    return round(price * (1 - discount_percent / 100), 2)

# description, inputs, and what we expect ("ValueError" = should be rejected)
TEST_CASES = [
    ("expected: half price",   100.0,  50, 50.0),
    ("boundary: full discount",100.0, 100,  0.0),
    ("invalid: negative price", -5.0,  10, "ValueError"),
]

def run_tests(cases):
    passed = 0
    for description, price, discount, expected in cases:
        try:
            got = final_price(price, discount)
        except ValueError:
            got = "ValueError"
        ok = (got == expected)
        passed += ok
        print(f"{'PASS' if ok else 'FAIL':<6}{description}")
    print(f"{passed}/{len(cases)} passed")

run_tests(TEST_CASES)

In your assessment

Keep your cases in one list and run them after every change. Paste the final report into your evidence, and explain one boundary and one invalid case in words so the marker sees you understood why each was tested.

Why it matters at Level 3: a self-testing program demonstrates the organised, comprehensive testing the refined (Excellence) criterion asks for — and shows real software-development practice.

Showing a bug fixedL2L3

Debugging is evidence — if you capture it.

What it is

A short before-and-after record: the test that failed, what was wrong, the change you made, and the test passing afterwards. Both standards reward "testing and debugging" — and the debugging only counts if it is visible.

Why it matters

It is the clearest possible proof of refinement. A marker reading "input 100 returned Merit, should be Excellence; fixed the threshold from > to >=; retested, now correct" can see genuine iterative improvement — exactly what Excellence rewards.

Example — a debugging note

# BUG found by boundary test:
#   grade_mark(80) returned "Merit", expected "Excellence".
#   Cause: threshold used  if mark > 80  (strictly greater).
#   Fix:   changed to       if mark >= 80
#   Retested boundary cases 79, 80, 81 -> Merit, Excellence, Excellence. Correct.

A handful of notes like this through your project tells the whole story of how the program got better.

In your assessment

Each time a test fails, capture the four parts: the failing case, the cause, the fix, the passing retest. Screenshots, a short log, or commit messages all work. This is also where version history (e.g. Git commits) becomes evidence rather than admin.

Common gotcha

Fixing bugs silently as you go and keeping no record. The work happened, but you cannot prove it — so it cannot lift your grade. Leave the trail.

Putting it togetherone task, two grades

Achieved → Excellence, on one small function

The same postage calculator, before and after the work this guide is about.

Two versions of one function do the identical job. Version 1 works on expected input and nothing else — magic numbers, no validation, a crash waiting for the first odd input. Version 2 is the same logic made flexible and robust: named constants, validated input, the invalid case handled, clear structure. Run them side by side and you can see precisely where the grade difference lives.

What changed between the two:

  • bare numbers like 500 and 20 became named constants;
  • input is validated for type and range before it is used;
  • invalid weights are rejected with a clear error instead of a wrong answer or a crash;
  • the bands are derived from the constants, so the program flexes when they change.

None of that made the program cleverer. It made it robust — and robust is what the top of the standard asks for.
